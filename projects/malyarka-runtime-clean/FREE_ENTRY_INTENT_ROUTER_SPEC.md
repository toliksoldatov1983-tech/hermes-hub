# FREE_ENTRY_INTENT_ROUTER_SPEC

## 1. Цель

Описать будущую архитектуру, в которой пользователь может войти в любую дверь проекта «Малярка» через свободную речь, без запоминания точных команд. Бот понимает смысл фразы, направляет её в нужный режим, а при неуверенности — уточняет, а не угадывает.

Документ также описывает:
- DeepSeek как резервный AI-провайдер Hermes;
- независимое ревью (делает Codex/Hermes — проверяет DeepSeek — решает пользователь);
- запреты безопасности.

На этом этапе документ — архитектурная спецификация. Код проекта не меняется.

---

## 2. Свободный вход (Free Entry)

### 2.1. Проблема

Сейчас пользователь должен явно указать команду (`/заказ`, `/идеи`, `/инженер`, `/админ`) или нажать кнопку, чтобы попасть в нужную дверь. Если пользователь отправил фразу без команды, система либо показывает меню выбора (в `neutral`), либо отвечает заглушкой.

Проблема: пользователь не обязан помнить команды и кнопки. Он хочет сказать «прими заказ», «давай обсудим новую идею», «проверь текущий статус проекта» — и бот сам понимает, куда направить.

### 2.2. Решение

Добавить слой **Free Entry** перед существующим router/state machine. Этот слой перехватывает любую свободную фразу пользователя в режиме `neutral` (когда режим не выбран) и пытается определить намерение (intent).

Последовательность:

```text
Свободная фраза пользователя
        │
        ▼
┌───────────────────┐
│  Free Entry       │
│  Intent Dispatcher│ ← определяет намерение
└───────┬───────────┘
        │
        ├── намерение ясное и уверенное ──→ маршрутизация в дверь
        │
        ├── намерение неясное ──→ уточняющий вопрос
        │
        └── намерение пустое/не определено ──→ главное меню
```

### 2.3. Как определить намерение

Dispatcher использует три подхода, перечисленные по приоритету:

**A. Локальные сигнатуры (приоритет 1, без AI)**

Проверка текста по локальным сигнатурам намерений без вызова AI:

- Фраза содержит размеры (`1000*400`, `500 x 300`, `2000 800 2`) → намерение `order`
- Фраза содержит ключевые слова-триггеры:
  - `заказ`, `размер`, `фасад`, `деталь`, `сделай`, `прими`, `новый заказ` → намерение `order`
  - `идея`, `давай обсудим`, `подумать`, `как думаешь`, `а что если`, `предлагаю` → намерение `ideas`
  - `статус`, `что сделано`, `следующий шаг`, `проверь`, `отчёт`, `задача`, `этап` → намерение `engineer`
  - `админ`, `диагностика`, `проверь режим`, `сброс` → намерение `admin`

- Если ни одна сигнатура не совпала → возвращается `intent=unknown`, система переходит к AI-распознаванию (B) или уточнению.

**B. DeepSeek NLU (приоритет 2, локальный AI без вызова API)**

Локальный AI-классификатор на базе DeepSeek (пока без внешнего API):

- получает текст;
- возвращает: `intent`, `confidence`, `reason`;
- конфиденциальность: не читает ключи, не вызывает внешний API;
- если `confidence < порога` → `intent=unknown`.

Порог уверенности задаётся в конфигурации. По умолчанию: `0.7`.

**C. Уточняющий вопрос (приоритет 3)**

Если оба подхода (A и B) не дали уверенного намерения, система задаёт уточняющий вопрос:

```text
Я не совсем понял, что вы хотите сделать. Выберите:

🧾 Новый заказ
💬 Идея / обсуждение
🛠 Инженер проекта
🔧 Админ (служебный)
← Отмена
```

После выбора пользователь направляется в соответствующую дверь.

## 3. Диспетчер намерений (Intent Dispatcher)

### 3.1. Структура

Dispatcher — это отдельный модуль, который:

1. Получает свободный текст.
2. Пытается определить намерение (через A → B → C).
3. Если намерение определено уверенно — маршрутизирует.
4. Если не уверен — уточняет.
5. Если отказался (Отмена) — возвращает в главное меню.

### 3.2. Контракт входа/выхода

**Вход:**
```text
text: str          — сырой текст пользователя
mode: str          — текущий режим (neutral / order / chat / engineer / admin)
confidence_threshold: float = 0.7
```

**Выход:**
```text
{
  "intent": "order" | "ideas" | "engineer" | "admin" | "unknown",
  "confidence": float,        // 0.0 — 1.0
  "method": "signature" | "ai" | "clarification" | "none",
  "clarification_needed": bool,
  "clarification_question": str | null,
  "action": "route" | "ask" | "menu",
}
```

### 3.3. Поведение в зависимости от текущего режима

| Текущий режим | Действие при свободной фразе                              |
|---------------|-----------------------------------------------------------|
| `neutral`     | Запустить Intent Dispatcher                               |
| `order`       | Игнорировать Dispatcher — текст идёт в парсер заказа      |
| `ideas`       | Игнорировать Dispatcher — текст идёт как свободный разговор |
| `engineer`    | Игнорировать Dispatcher — текст идёт в карточку задачи    |
| `admin`       | Игнорировать Dispatcher — текст идёт как команда админа   |

Dispatcher срабатывает **только** в `neutral` mode. В остальных режимах текст уже направлен в соответствующую дверь существующим router/state machine.

## 4. Четыре двери

### 4.1. `/заказ` (Order)

**Вход:** размеры, описание заказа.

**Назначение:** строгий производственный режим приёма заказов.

**Поведение:**
- принимает размеры в форматах `1000*400`, `1000 x 400`, `1000 400` и др.;
- разбирает строки, находит спорные строки;
- показывает preview;
- готовит Excel для Corel;
- готовит Файл Малярки;
- не болтает свободно;
- не додумывает размеры;
- не меняет правила и код.

**Откуда направляется:** из Free Entry Dispatcher при `intent=order`, а также по прямой команде `/заказ`.

### 4.2. `/идеи` (Ideas)

**Вход:** свободный текст, предложения, обсуждения.

**Назначение:** свободный разговор, генерация идей, подготовка выжимки для инженера.

**Поведение:**
- обсуждает идеи, оптимизации, бизнес-вопросы;
- не парсит размеры как заказ;
- не меняет код и файлы проекта;
- может подготовить выжимку (summary) для `/инженер`.

**Откуда направляется:** из Free Entry Dispatcher при `intent=ideas`, а также по прямой команде `/идеи`.

### 4.3. `/инженер` (Engineer)

**Вход:** описание задачи, запрос статуса, команда проверки.

**Назначение:** управление проектом — задачи, проверки, отчёты, принятие этапов.

**Поведение:**
- показывает текущий статус проекта;
- формирует карточку задачи;
- подготавливает задание для Codex/Hermes;
- проверяет отчёты;
- требует подтверждение перед изменениями;
- не меняет код напрямую.

**Откуда направляется:** из Free Entry Dispatcher при `intent=engineer`, а также по прямой команде `/инженер`.

### 4.4. `/админ` (Admin, скрытый)

**Вход:** команды диагностики.

**Назначение:** служебная диагностика, доступна только владельцу.

**Поведение:**
- показывает безопасный статус бота;
- проверяет текущий режим;
- сбрасывает режим;
- не выводит секреты, токены, `.env`.
- **Не показывается** в главном меню и в Dispatcher в явном виде.

**Доступ:** только через прямую команду `/админ` или точное совпадение с сигнатурой `admin`.

**Dispatcher не должен маршрутизировать случайные фразы в `/админ`.** Если пользователь случайно написал что-то похожее на команду админа, Dispatcher должен уточнить.

## 5. Как бот понимает смысл фразы (NLU)

### 5.1. Уровни понимания

**Уровень 1 — Локальные сигнатуры (без AI)**

- Проверка по ключевым словам и паттернам.
- Проверка по формату (наличие размеров / цифр с разделителями).
- Мгновенно, не требует сети, не вызывает API.
- Не может распознать сложные намерения.

**Уровень 2 — AI-классификатор (локальный DeepSeek, без внешнего API)**

- Использует DeepSeek-модель в режиме ограниченного классификатора.
- Принимает текст, возвращает `intent`, `confidence`, `reason`.
- Explicit prompt запрещает додумывать, угадывать, менять текст.
- Не вызывает внешний API.
- Не читает ключи.

**Уровень 3 — Уточнение**

- Если уверенность < порога — задаёт уточняющий вопрос.
- Пользователь выбирает из предложенных вариантов.
- После выбора — маршрутизация.

**Уровень 4 — Прямая команда**

- `/заказ`, `/идеи`, `/инженер`, `/админ`.
- Всегда имеет наивысший приоритет, игнорирует Dispatcher.

### 5.2. Примеры распознавания

| Фраза пользователя                        | Определённое намерение | Метод       | Уверенность |
|-------------------------------------------|------------------------|-------------|-------------|
| `1000*400`                                | `order`               | signature   | 1.0         |
| `500 x 300 2`                             | `order`               | signature   | 1.0         |
| `прими заказ`                             | `order`               | signature   | 0.95        |
| `давай обсудим новую идею`                | `ideas`               | signature   | 0.9         |
| `а что если добавить Excel-выгрузку`      | `ideas`               | ai          | 0.8         |
| `какой статус проекта`                    | `engineer`            | signature   | 0.95        |
| `проверь следующий шаг`                   | `engineer`            | signature   | 0.9         |
| `как думаешь, стоит ли делать архив`      | `ideas`               | ai          | 0.75        |
| `сделать файл малярки из заказа`          | `engineer`            | ai          | 0.7         |
| `погода сегодня хорошая`                  | `unknown`             | none        | 0.0         |
| `привет`                                  | `unknown`             | none        | 0.0         |
| `админ статус`                            | (уточнение)           | —           | —           |

### 5.3. Explicit prompt для AI-классификатора

```
Ты — классификатор намерений проекта «Малярка».
Определи намерение пользователя по тексту.

Доступные намерения:
- order: пользователь хочет оформить, обсудить или изменить заказ фасадов. Фраза содержит размеры, материалы, описание фасада, просьбу принять заказ.
- ideas: пользователь хочет обсудить идею, предложение, улучшение. Фраза содержит «давай», «а что если», «подумать», «идея», предложение по улучшению.
- engineer: пользователь хочет узнать статус проекта, проверить отчёт, сформировать задачу, управлять проектом.
- unknown: намерение не определено.

Запрещено:
- додумывать;
- исправлять текст пользователя;
- угадывать размеры;
- менять намерение, если не уверен.

Верни только JSON:
{"intent": "...", "confidence": 0.0-1.0, "reason": "..."}
```

## 6. Когда уточнять, а не угадывать

### 6.1. Правила уточнения

Система обязана уточнить (не угадывать) в следующих случаях:

1. **Уверенность < порога** (`confidence < 0.7`).
2. **Фраза выглядит как команда админа**, но не является точной `/админ` — уточнить во избежание случайного входа в скрытый режим.
3. **Фраза может быть интерпретирована в два разных намерения** с разницей уверенности менее 0.15 — уточнить.
4. **Фраза пустая или состоит только из стоп-слов** (привет, спасибо, ок, да, нет, пока) — уточнить, показать главное меню.
5. **Фраза содержит опасные подстроки** (`.env`, `token`, `orders.db`, `bot.py`, `git add`, `commit`, `push`) — НЕ уточнять, а сразу блокировать с сообщением безопасности.

### 6.2. Формат уточнения

```text
Я не уверен, что именно вы хотите сделать.

Вы сказали: «{текст}»

Выберите действие:
🧾 Разобрать как заказ
💬 Обсудить как идею
🛠 Передать инженеру
← Отмена
```

Если намерение частично определено (например, `intent=order` с `confidence=0.6`), порядок кнопок меняется так, чтобы наиболее вероятное намерение было первым.

Если есть риск скрытого режима админа, кнопка админа **не показывается** — только уточнение через другие двери или отмена.

### 6.3. Запрет на угадывание

Система **никогда** не должна:
- автоматически создавать заказ из неясной фразы;
- отправлять фото во внешние AI/API без явного выбора;
- запускать Hermes/Codex/DeepSeek API без явного выбора;
- менять файлы, код, правила на основе неясной фразы;
- входить в `/админ` по неясной фразе.

## 7. DeepSeek как резервный провайдер Hermes

### 7.1. Текущая ситуация

Hermes настроен на `deepseek-reasoner` как основной провайдер. Резервный провайдер (`fallback_model`) в `config.yaml` не настроен.

Если GPT-5.5 / OpenAI Codex (или текущий основной провайдер) недоступен, нестабилен или отклоняет запрос — система остаётся без резерва.

### 7.2. Решение: DeepSeek как второй слой

DeepSeek становится вторым слоем (fallback provider) в цепочке Hermes.

Схема:

```text
Пользовательская задача
        │
        ▼
┌──────────────────────┐
│  Основной провайдер   │ ← GPT-5.5 / OpenAI Codex / deepseek-reasoner
│  (Primary)            │
└──────┬───────────────┘
       │
       ├── успех ──→ результат пользователю
       │
       └── ошибка / таймаут / отказ ──→ fallback
                                         │
                                         ▼
                               ┌──────────────────────┐
                               │  DeepSeek (Fallback)  │
                               │  deepseek-v4-flash    │
                               └──────────────────────┘
                                         │
                                         ├── успех ──→ результат пользователю
                                         │
                                         └── ошибка ──→ сообщение о недоступности
```

### 7.3. Конфигурация fallback_model в Hermes

```yaml
# config.yaml (фрагмент)
fallback_model:
  provider: deepseek
  model: deepseek-v4-flash
  base_url: https://api.deepseek.com/v1
```

### 7.4. Правила fallback

- Fallback срабатывает автоматически при:
  - rate limit (429);
  - service unavailable (503);
  - overload (529);
  - connection timeout / network error;
  - отказ модели выполнить запрос.
- Fallback **не срабатывает** при:
  - отказе по безопасности (неверный ключ, запрещённый токен);
  - ошибке в запросе (invalid request);
  - превышении лимита токенов, если это не временная ошибка.
- При срабатывании fallback пользователь видит сообщение:
  ```
  [Основной AI-провайдер временно недоступен.
  Использован резервный провайдер DeepSeek.]
  ```
- При ошибке обоих провайдеров:
  ```
  [AI-сервисы временно недоступны.
  Попробуйте позже.]
  ```

### 7.5. Безопасность DeepSeek-провайдера

- `DEEPSEEK_API_KEY` передаётся только через `.env` или `config.yaml` как API key.
- Не выводится в диагностику.
- Не логируется.
- Не передаётся в Telegram-бот.
- `base_url` проверяется на HTTPS.

## 8. Независимое ревью

### 8.1. Проблема

Сейчас Codex/Hermes делает изменения, и пользователь сам проверяет результат. Нет автоматического или полуавтоматического второго мнения.

### 8.2. Решение

Трёхуровневое ревью:

```text
┌─────────────────────────────────────────────────────┐
│  1. Codex / Hermes                                   │
│     Делает: код, файлы, изменения, тесты              │
│     Показывает: diff, отчёт, результат проверок      │
└──────────────────────┬──────────────────────────────┘
                       │
                       ▼
┌─────────────────────────────────────────────────────┐
│  2. DeepSeek / другой провайдер (Reviewer)           │
│     Проверяет:                                        │
│     - логику изменений                                │
│     - риски                                           │
│     - соответствие правилам проекта (AGENTS.md)       │
│     - безопасность                                    │
│     Возвращает: review report:                        │
│       - статус: approve / reject / needs_changes      │
│       - замечания                                     │
│       - рекомендации                                  │
└──────────────────────┬──────────────────────────────┘
                       │
                       ▼
┌─────────────────────────────────────────────────────┐
│  3. Пользователь                                     │
│     Принимает решение:                                │
│     - Принять (approve)                               │
│     - Отклонить (reject)                              │
│     - Отправить на доработку (rework)                  │
└─────────────────────────────────────────────────────┘
```

### 8.3. Как это работает

**Шаг 1. Codex/Hermes выполняет задачу.**

```
Codex/Hermes:
- создал файл module.py
- изменил bot.py
- запустил pytest: 12 passed
- git diff показал изменения
```

**Шаг 2. DeepSeek проверяет.**

```
Reviewer (DeepSeek):
- проверка логики: OK
- AGENTS.md compliance: OK
- безопасность: OK — .env, orders.db, bot.py не трогались
- риски: нет
- статус: approve
```

**Шаг 3. Пользователь решает.**

```
Пользователь:
- прочитал отчёт Codex/Hermes;
- прочитал review DeepSeek;
- решение: Принять / Отклонить / Доработать.
```

### 8.4. Prompt для Review Agent

```
Ты — независимый ревьюер проекта «Малярка».
Проверь отчёт Codex/Hermes на:

1. Безопасность:
   - .env, orders.db, .venv, .git, bot.py не трогались;
   - токены, ключи, пароли не выводились;
   - polling/gateway не запускались;
   - git add/commit/push не выполнялись.

2. Соответствие AGENTS.md:
   - не нарушены ли ограничения;
   - не менялись бизнес-правила без разрешения;
   - не додумывались размеры, материал, цвет, покрытие.

3. Логика:
   - изменения соответствуют задаче;
   - нет противоречий с MALYARKA_CURRENT_STATE.md.

4. Тесты:
   - тесты запускались;
   - результат тестов (passed/failed);
   - покрытие критических путей.

Верни JSON:
{
  "status": "approve" | "reject" | "needs_changes",
  "safety_ok": true | false,
  "rules_ok": true | false,
  "logic_ok": true | false,
  "tests_ok": true | false,
  "issues": ["описание проблемы 1", ...],
  "recommendations": ["рекомендация 1", ...]
}
```

### 8.5. Автоматизация ревью

На первом этапе ревью — **полуавтоматическое**:

- Codex/Hermes выполняет задачу и показывает отчёт.
- Пользователь вручную запускает ревью через команду `!review` или кнопку в Telegram.
- DeepSeek проверяет и показывает review report.
- Пользователь принимает решение.

На будущих этапах (после отдельного решения):

- ревью запускается автоматически после каждого изменения;
- review report прикрепляется к карточке задачи в `/инженер`;
- если ревью `reject`, задача автоматически отправляется на доработку.

### 8.6. Критерии ревью

|---------------------------|----------------------------------|------------|

| Соответствие AGENTS.md    | Правила проекта соблюдены        | Да         |
| Логика                    | Изменения соответствуют задаче   | Да         |
| Тесты                     | Все тесты проходят               | Нет (но предупреждение) |
| Нет додумывания           | Размеры/цвета/материалы не угаданы | Да      |
| Нет изменения bot.py      | Старый bot.py не трогался        | Да         |
