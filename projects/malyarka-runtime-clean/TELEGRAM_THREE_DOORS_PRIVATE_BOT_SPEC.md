# TELEGRAM_THREE_DOORS_PRIVATE_BOT_SPEC

## 1. Цель

Описать личный Telegram-бот «Малярка» с тремя видимыми дверями и одним скрытым служебным режимом.

Бот нужен как единая личная точка входа для владельца:

- рабочий заказ;
- управление проектом;
- свободное обсуждение идей;
- скрытая диагностика.

На этом этапе это спецификация. Код проекта не меняется.

## 2. Главная модель: один бот, три видимые двери

В Telegram существует один личный бот «Малярка».

Видимые режимы:

1. `/заказ` — рабочий режим.
2. `/инженер` — управление проектом.
3. `/идеи` — свободный разговор.

Скрытый режим:

4. `/админ` — служебный режим только для владельца.

Пользователь не обязан помнить команды. Основной вход — кнопки главного меню и подсказки. Команды остаются быстрым способом переключения.

## 3. Личный доступ

Бот доступен только владельцу по Telegram `owner_id`.

Правила доступа:

- все режимы на первом этапе доступны только владельцу;
- чужие `user_id` игнорируются или получают короткий отказ;
- группы и групповые чаты игнорируются;
- никакие рабочие функции не выполняются для чужих пользователей;
- `owner_id` не выводится публично, кроме скрытого `/админ` для владельца.

## 4. Главное меню кнопками

Главное меню показывает три видимые двери:

- 🧾 Новый заказ
- 🛠 Инженер проекта
- 💬 Идеи / разговор

Соответствие кнопок режимам:

- 🧾 Новый заказ → `/заказ`
- 🛠 Инженер проекта → `/инженер`
- 💬 Идеи / разговор → `/идеи`

Скрытый `/админ` в главном меню не показывать.

## 5. Команды и подсказки

Команды остаются быстрым способом:

- `/заказ`
- `/инженер`
- `/идеи`
- скрытая `/админ`

Но основной сценарий должен работать без запоминания команд:

- через кнопки;
- через подсказки;
- через выбор действия, если бот не понимает намерение.

## 6. Поведение без выбранного режима: текст с размерами

Если владелец отправил текст с размерами без активного режима, бот не должен молча угадывать режим.

Бот показывает выбор:

- Разобрать как заказ
- Обсудить как идею
- Передать в инженерную задачу
- Отмена

После выбора:

- «Разобрать как заказ» — обработать текст через режим `/заказ`;
- «Обсудить как идею» — перейти в `/идеи` и обсудить текст;
- «Передать в инженерную задачу» — подготовить карточку задачи для `/инженер`;
- «Отмена» — ничего не менять и показать главное меню.

## 7. Поведение без выбранного режима: фото или скриншот

Если владелец отправил фото/скриншот без активного режима, бот не должен отправлять фото во внешние сервисы и не должен угадывать намерение.

Бот показывает выбор:

- Разобрать как заказ
- Обсудить как идею
- Передать в инженерную задачу
- Отмена

После выбора:

- «Разобрать как заказ» — если фото-режим не подключён, ответить безопасной заглушкой;
- «Обсудить как идею» — перейти в `/идеи` без внешнего анализа фото;
- «Передать в инженерную задачу» — подготовить задачу без отправки фото наружу;
- «Отмена» — ничего не менять.

## 8. Фото/Vision

На текущем этапе фото-режим не подключён.

Если пользователь выбрал «Разобрать как заказ» для фото, бот отвечает:

```text
Фото-режим ещё не подключён. Отправьте размеры текстом.
```

Правила безопасности:

- фото не отправлять во внешние AI/API без отдельного разрешения владельца;
- Vision не подключать без отдельного этапа;
- DeepSeek API не подключать на этом этапе;
- фото не использовать для автоматического заказа, пока режим не принят отдельно.

## 9. Режим `/заказ`

Назначение: рабочий производственный режим.

Режим `/заказ` должен:

- разбирать размеры;
- соблюдать правило: первое число — высота, второе — ширина, третье — количество;
- если количество не указано, считать `quantity = 1`;
- находить спорные строки;
- делать preview заказа;
- готовить Excel для Corel;
- в будущем готовить Файл Малярки;
- в будущем готовить архив;
- не включать спорные строки в чистовой результат;
- не создавать финальные файлы, пока есть спорные строки.

Режим `/заказ` не должен:

- болтать свободно;
- менять код;
- менять проектные правила;
- запускать внешние AI/API;
- работать с фото как с Vision без отдельного разрешения.

## 10. Режим `/инженер`

Назначение: управление проектом «Малярка».

Режим `/инженер` должен:

- показывать текущий статус проекта;
- показывать следующий безопасный шаг;
- готовить задачи для Hermes;
- проверять отчёты;
- помогать принимать или отклонять этапы;
- переводить идеи в чёткие карточки задач;
- требовать подтверждение перед любыми изменениями.

Правило: любые изменения только через карточку задачи и подтверждение владельца.

Режим `/инженер` не должен:

- самовольно менять файлы;
- читать `.env`;
- менять `orders.db`;
- трогать `.git`, `.venv`, старый `bot.py`;
- запускать polling/gateway без отдельного разрешения;
- делать `git add`, commit, push.

## 11. Режим `/идеи`

Назначение: свободный разговор и обсуждение улучшений.

Режим `/идеи` может:

- обсуждать идеи и оптимизации;
- помогать думать о производстве, заказах, фасадах, Excel, CorelDRAW, Telegram UX;
- делать выжимку для передачи в `/инженер`;
- предлагать варианты будущих задач.

Режим `/идеи` не должен:

- парсить размеры как рабочий заказ;
- создавать финальные файлы;
- менять код;
- менять файлы проекта;
- запускать API, polling или gateway.

Если в `/идеи` пользователь прислал размеры, бот может обсудить идею, но не должен автоматически оформлять заказ без явного выбора «Разобрать как заказ».

## 12. Скрытый режим `/админ`

Назначение: безопасная диагностика только для владельца.

Правила:

- `/админ` доступен только владельцу;
- не показывать `/админ` в главном меню;
- чужим пользователям не раскрывать наличие режима;
- режим не должен выводить секреты, токены, ключи и `.env`.

Функции `/админ`:

- показать безопасный статус бота;
- проверить текущий режим;
- сбросить режим в нейтральный;
- проверить `state_version`;
- проверить, совпадает ли текущий пользователь с `owner_id`, без раскрытия секретов чужим пользователям;
- показать, что polling/gateway/API не запускались внутри тестового сценария, если это применимо.

## 13. Карточка задачи для `/инженер`

Карточка задачи должна иметь структуру:

```text
ЗАДАЧА:

ОЖИДАЕМЫЙ РЕЗУЛЬТАТ:

МОЖНО:

НЕЛЬЗЯ:

ПРОВЕРКИ:

СТАТУС:
```

Кнопки карточки:

- Разрешаю
- Отмена
- Исправить задачу
- Принять этап
- Отклонить

Правила карточки:

- без подтверждения «Разрешаю» изменения не выполняются;
- «Отмена» закрывает задачу без изменений;
- «Исправить задачу» возвращает к редактированию формулировки;
- «Принять этап» доступно только после отчёта и проверок;
- «Отклонить» фиксирует, что этап не принят.

## 14. Запреты

На этом и будущих безопасных этапах запрещено без отдельного прямого разрешения:

- читать `.env`;
- выводить `BOT_TOKEN`;
- читать или выводить токены, ключи, пароли;
- менять `orders.db`;
- трогать `.git`;
- трогать `.venv`;
- менять старый `bot.py`;
- запускать polling;
- запускать gateway;
- делать `git add`;
- делать commit;
- делать push;
- подключать DeepSeek API;
- подключать Vision;
- отправлять фото во внешние сервисы;
- менять бизнес-правила без подтверждения владельца.

## 15. Потенциальные файлы будущего кодового этапа

На будущем кодовом этапе потенциально могут понадобиться изменения в отдельных файлах новой модульной архитектуры, например:

- `malyarka_telegram/modes.py` — режимы и имена режимов;
- `malyarka_telegram/router.py` — маршрутизация команд, кнопок и сообщений;
- `malyarka_telegram/session.py` — хранение текущего режима;
- `malyarka_telegram/handlers.py` — обработка текста, фото и callback-кнопок;
- `malyarka_telegram/keyboards.py` — главное меню и кнопки выбора действия;
- `malyarka_telegram/messages.py` — тексты сообщений и подсказок;
- `tests/test_malyarka_telegram_*.py` — тесты Telegram-логики;
- при отдельном разрешении: `malyarka_telegram/app.py` для подключения к live-слою.

Ограничение: старый `bot.py` не менять без отдельного прямого разрешения.
