\# AGENTS.md — правила проекта «Малярка» для Codex



\## Роль Codex



Codex — рабочий coding-agent проекта «Малярка».



Codex может:

\- анализировать код;

\- предлагать архитектуру;

\- создавать новые модули;

\- писать тесты;

\- запускать проверки;

\- показывать diff;

\- готовить изменения для проверки пользователем.



Codex не является главным центром управления проектом.

Главный центр управления — мастер-выжимка проекта «Малярка» и чат инженера проекта.



\## Главные ограничения



Нельзя:

\- трогать .env;

\- трогать orders.db без отдельного разрешения;

\- трогать .venv;

\- трогать .git;

\- загружать токены, ключи, BOT\_TOKEN, OPENAI\_API\_KEY;

\- использовать git add .;

\- коммитить без разрешения пользователя;

\- пушить без разрешения пользователя;

\- менять боевой bot.py без отдельного разрешения;

\- принимать спорные строки как правильные;

\- угадывать размеры, количество, материал, цвет, покрытие или фрезеровку;

\- менять бизнес-правила без подтверждения пользователя.



\## Текущий статус проекта



Стабильная точка отката:

b453645 “Parse photo caption order fields”.



Текущую неудачную правку Vision не принимать и не коммитить.



Старый bot.py использовать только как источник уже рабочих частей.

Новую архитектуру собирать модульно, отдельно от старого bot.py.



\## Цель первой версии нового ядра



Сначала собрать стабильное ядро без фото/Vision:



1\. Заказ.

2\. Детали.

3\. Спорные строки.

4\. Расчёт площади.

5\. Файл Corel.

6\. Файл для малярки.

7\. SQLite.

8\. Архив.

9\. Справочники.

10\. Безопасность.

11\. Pytest-тесты.



Vision, склад, зарплаты, расходы, производственные статусы, голос и расширенную автоматизацию добавлять позже.



\## Правила размеров



Первое число — высота.

Второе число — ширина.

Третье число — количество.



Если количество не указано — quantity = 1.



Если высота, ширина, количество, примечание или смысл строки неясны — строка спорная.



Спорные строки нельзя включать в чистовой результат.

Пока есть спорные строки, финальные файлы создавать нельзя.



\## Свойства детали



Свойства детали хранить раздельно:



\- тип детали;

\- материал;

\- толщина;

\- высота;

\- ширина;

\- количество;

\- цвет;

\- покрытие;

\- тип фрезеровки;

\- сторона фрезеровки;

\- сторона покраски;

\- обработка торца;

\- группа;

\- подгруппа;

\- статус;

\- примечание;

\- источник строки;

\- уверенность распознавания.



Нельзя смешивать эти свойства в одно поле.



\## Corel-файл



Формат Corel:



1\. Высота

2\. Ширина

3\. Количество



Правила:

\- без заголовков;

\- первая строка пустая;

\- только подтверждённые строки;

\- без спорных строк;

\- без материала, цвета, покрытия, фрезеровки, площади, примечаний и итогов.

\## SQLite



SQLite использовать как основное хранилище.



Нужно предусмотреть:

\- orders;

\- order\_items;

\- disputed\_items;

\- materials;

\- colors;

\- coatings;

\- milling\_types;

\- prices;

\- files;

\- audit\_log;

\- settings;

\- command\_aliases.



Актуальное состояние заказа хранить отдельно от истории изменений.

Историю изменений писать в audit\_log.

Физически детали не удалять — использовать статус deleted/cancelled.



\## Проверки



После изменений запускать:



python -m py\_compile bot.py



Если появились тесты:



pytest -q



Всегда показывать:



git status --short --ignored



\## Формат ответа Codex



После работы Codex должен показать:



1\. Что сделал.

2\. Какие файлы изменил.

3\. Какие проверки запустил.

4\. Результат проверок.

5\. Что осталось сделать.

6\. Есть ли риск сломать старую рабочую часть.

7\. Нужен ли commit.
