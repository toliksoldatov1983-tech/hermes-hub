# MALYARKA_FILE_SPEC

## 1. Статус документа

Документ описывает будущий «Файл Малярки» как второй производственный экспорт после Excel для Corel.

Текущий этап: спецификация.

На этом этапе код проекта не меняется.

## 2. Назначение Файла Малярки

Файл Малярки — это производственный файл для работы малярки.

Назначение:

- дать малярке понятную таблицу по заказу;
- показать размеры и количество деталей;
- показать площадь одной детали и площадь строки;
- показать общую подтверждённую площадь;
- показать спорные строки отдельно или заблокировать чистовой выпуск;
- добавить подтверждённые производственные поля: материал, цвет, толщина, тип работы, фрезеровка, примечания;
- добавить цены и суммы только если они подтверждены.

Файл Малярки не является:

- файлом для CorelDRAW;
- заменой Excel для Corel;
- архивом заказа;
- базой данных;
- инструментом для угадывания материала, цвета, цены или смысла спорных строк.

## 3. Отличие от Excel для Corel

Excel для Corel уже работает и не должен меняться из-за Файла Малярки.

Excel для Corel:

- предназначен для CorelDRAW;
- содержит только чистые строки `height / width / quantity`;
- не содержит заголовков;
- не содержит площади;
- не содержит материала, цвета, покрытия, цены, примечаний;
- не содержит спорных строк;
- остаётся отдельным экспортом.

Файл Малярки:

- предназначен для производства и малярки;
- может иметь заголовки колонок;
- содержит размеры, количество и площадь;
- содержит общую подтверждённую площадь;
- может содержать подтверждённые материал, цвет, толщину, тип работы, фрезеровку, примечания;
- может содержать цену за м², сумму строки и общую сумму только если цена подтверждена;
- показывает спорные строки отдельно или блокирует финальный чистовой файл.

Главное правило: Файл Малярки не должен менять формат Excel для Corel.

## 4. Минимальные колонки первого этапа

Первый кодовый этап Файла Малярки должен использовать минимальный безопасный набор колонок.

Колонки:

1. `row_number` / № строки
2. `height` / высота
3. `width` / ширина
4. `quantity` / количество
5. `item_area_m2` / площадь одной детали, м²
6. `row_area_m2` / площадь строки, м²
7. `row_status` / статус строки: `confirmed` или `disputed`
8. `raw_text` / исходный текст строки
9. `dispute_reason` / причина спорности, если есть

Правила:

- для подтверждённой строки должны быть высота, ширина и количество;
- для спорной строки размеры не угадывать;
- если поле неизвестно, оно должно быть пустым или явно не заполненным, а не выдуманным;
- статус строки обязателен.

## 5. Дополнительные поля только если подтверждены

Дополнительные поля можно добавлять только если данные явно подтверждены пользователем или надёжно получены из принятого источника.

Поля:

- `material` / материал;
- `color` / цвет;
- `thickness` / толщина;
- `work_type` / тип работы;
- `milling_type` / тип фрезеровки, если требуется;
- `milling_side` / сторона фрезеровки, если требуется;
- `paint_side` / сторона покраски, если требуется;
- `edge_processing` / обработка торца, если требуется;
- `note` / примечание;
- `price_per_m2` / цена за м²;
- `row_total` / сумма строки;
- `order_total` / общая сумма.

Запрещено:

- угадывать материал;
- угадывать цвет;
- угадывать толщину;
- угадывать тип работы;
- угадывать фрезеровку;
- угадывать цену;
- считать сумму, если цена не подтверждена.

## 6. Правила площади

Площадь считается в квадратных метрах.

Формулы:

```text
площадь одной детали = height * width / 1_000_000
площадь строки = площадь одной детали * quantity
```

Где:

- `height` — высота в миллиметрах;
- `width` — ширина в миллиметрах;
- `quantity` — количество деталей;
- первое число в заказе — высота;
- второе число — ширина;
- третье число — количество;
- если количество не указано, `quantity = 1`.

Итоги:

- итоговая площадь считается только по подтверждённым строкам;
- спорные строки не включаются в чистовой итог;
- пустые, нулевые или отрицательные размеры не должны попадать в подтверждённый итог;
- округление площади должно быть стабильным и проверяемым тестами.

Рекомендуемое отображение:

- `item_area_m2` и `row_area_m2` показывать с понятной точностью, например до 3 знаков после запятой;
- внутренний расчёт лучше держать числом, а форматирование делать на этапе экспорта.

## 7. Правила спорных строк

Спорная строка — строка, где неясны размер, количество, смысл, материал, цвет, цена или другое обязательное поле для выбранного действия.

Правила:

- спорные строки показывать отдельно или в отдельном блоке файла;
- спорные строки не включать в чистовую итоговую площадь;
- спорные строки не включать в чистовую сумму;
- спорные строки не использовать для финального производственного результата без подтверждения;
- не угадывать размеры, материал, цвет, цену или тип работы;
- сохранять `raw_text`, чтобы пользователь мог исправить исходную строку;
- сохранять `dispute_reason`, чтобы было понятно, что нужно исправить.

Если в заказе есть спорные строки:

- финальный чистовой Файл Малярки должен быть заблокирован;
- или файл должен быть явно помечен как неполный/черновой;
- предпочтение для первого этапа: блокировать чистовой файл, чтобы не отдать в производство неполный заказ.

## 8. Формат будущего экспорта

Первый кодовый этап:

- формат: `.xlsx`;
- отдельный экспорт от Excel для Corel;
- файл с заголовками колонок;
- файл с итоговой подтверждённой площадью;
- без спорных строк в чистовом результате;
- при спорных строках — блокировка чистового файла или явная черновая пометка.

Позже:

- PDF-карточка заказа;
- архив заказа;
- привязка к SQLite;
- история изменений в `audit_log`;
- производственные статусы;
- расширенные поля цены, покрытия, фрезеровки и обработки торца.

## 9. Поведение для чистого заказа

Чистый заказ — заказ без спорных строк.

Для чистого заказа Файл Малярки должен:

- создаваться как `.xlsx`;
- содержать все подтверждённые строки;
- содержать площадь каждой детали;
- содержать площадь каждой строки;
- содержать общий итог по количеству деталей;
- содержать общий итог по площади;
- не менять Excel для Corel;
- не добавлять неподтверждённые поля.

## 10. Поведение для спорного заказа

Спорный заказ — заказ, где есть хотя бы одна спорная строка.

Для спорного заказа:

- чистовой Файл Малярки не создаётся;
- пользователь должен увидеть, какие строки спорные;
- подтверждённые строки можно показать в preview;
- итоговую площадь считать только по подтверждённым строкам и пометить как неполную;
- экспорт в производство должен быть заблокирован до исправления спорных строк.

## 11. Цена и суммы

Цена — опасное поле, потому что ошибка сразу влияет на деньги.

Правила:

- `price_per_m2` использовать только если цена подтверждена;
- `row_total = row_area_m2 * price_per_m2` считать только если подтверждены площадь и цена;
- `order_total` считать только по строкам с подтверждённой ценой;
- если цена не подтверждена, сумма строки и общая сумма не считаются;
- нельзя брать цену из догадки;
- нельзя подставлять цену по умолчанию без явного правила и подтверждения.

## 12. Связь с Telegram UX

На будущем Telegram-этапе Файл Малярки может получить отдельную кнопку, например:

- `Скачать Файл Малярки`

Правила UX:

- кнопка не должна заменять `Скачать Excel для Corel`;
- для чистого заказа могут быть две разные кнопки: Corel и Файл Малярки;
- при спорных строках кнопка чистового Файла Малярки не показывается или показывает отказ с причиной;
- основной путь исправления спорных строк остаётся через редактируемый текст.

## 13. Потенциальные файлы следующего кодового этапа

На следующем кодовом этапе потенциально могут понадобиться изменения или новые файлы:

- `malyarka_core/exports/malyarka_file.py` — новый экспорт Файла Малярки;
- `malyarka_core/exports/__init__.py` — экспорт функции, если нужно;
- `malyarka_core/models.py` или существующие модели — только если нужны явные структуры данных;
- `malyarka_telegram/keyboards.py` — будущая кнопка `Скачать Файл Малярки`;
- `malyarka_telegram/handlers.py` — будущая обработка действия скачивания;
- `malyarka_telegram/messages.py` — тексты отказов и подсказок;
- `tests/test_malyarka_file_export.py` — тесты нового экспорта;
- `tests/test_malyarka_telegram_messages.py` — тесты будущих кнопок, если Telegram-слой будет разрешён;
- `tests/test_malyarka_telegram_app_router_integration.py` — интеграционные тесты Telegram-слоя, если будет разрешён live-layer без polling.

Ограничения следующего этапа:

- старый `bot.py` не менять без отдельного прямого разрешения;
- `orders.db` не подключать без отдельного разрешения;
- `.env`, токены и ключи не читать;
- polling/gateway не запускать;
- Excel для Corel не менять в сторону производственного файла.

## 14. Тесты будущего кодового этапа

Нужны тесты:

1. Чистый заказ создаёт Файл Малярки.
2. Спорный заказ блокирует чистовой Файл Малярки.
3. Площадь одной детали считается правильно: `height * width / 1_000_000`.
4. Площадь строки считается правильно: `item_area_m2 * quantity`.
5. Итоговая площадь считается только по подтверждённым строкам.
6. Спорные строки не включаются в чистовой итог.
7. Excel для Corel не меняется и остаётся только `height / width / quantity`.
8. Пустой заказ блокируется.
9. Нулевые или отрицательные размеры блокируются.
10. Неподтверждённая цена не считается.
11. `row_total` считается только при подтверждённой цене.
12. `order_total` считается только по подтверждённым ценам.
13. Неподтверждённые материал, цвет, толщина и тип работы не подставляются.
14. При спорных строках сохраняются `raw_text` и `dispute_reason`.
15. Экспорт `.xlsx` создаётся в безопасной временной директории в тестах.
16. Polling не запускается в тестах.
17. Gateway не запускается в тестах.
18. Внешние AI/API не вызываются в тестах.
19. `.env` и `BOT_TOKEN` не читаются в тестах.

## 15. Критерии приёмки будущего кода

Кодовый этап можно принимать, если:

- появился отдельный модуль Файла Малярки;
- Excel для Corel не изменил формат;
- чистый заказ создаёт `.xlsx` Файл Малярки;
- спорный заказ блокирует чистовой экспорт;
- площадь считается тестами;
- цена не считается без подтверждения;
- тесты проходят;
- `py_compile` проходит для изменённых Python-файлов;
- polling/gateway/API не запускались;
- `.env`, `BOT_TOKEN`, `orders.db`, `.venv`, `.git`, старый `bot.py`, фото, токены, ключи и личные данные не трогались.

## 16. Безопасность и границы

Этот документ фиксирует только спецификацию.

В рамках текущего этапа запрещено:

- менять код проекта;
- менять `MALYARKA_CURRENT_STATE.md`;
- читать или менять `.env`;
- читать или выводить `BOT_TOKEN`;
- менять `orders.db`;
- трогать `.venv`;
- трогать `.git`;
- менять старый `bot.py`;
- запускать polling;
- запускать gateway;
- вызывать внешние AI/API;
- устанавливать пакеты;
- делать `git add`, commit или push.

## 17. Итоговое правило

Файл Малярки — это отдельный второй производственный экспорт.

Он должен помогать малярке видеть заказ, площадь и подтверждённые производственные данные.

Он не должен ломать или усложнять Excel для Corel, который остаётся чистым экспортом только `height / width / quantity`.
