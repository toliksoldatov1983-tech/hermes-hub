# TELEGRAM_MODES_SPEC

## 1. Цель

Один Telegram-бот должен поддерживать несколько режимов, чтобы не смешивать:

- строгую обработку заказов;
- свободное обсуждение;
- инженерные задачи;
- выжимки идей;
- просмотр правил.

Главная цель:

не допустить, чтобы разговорный режим случайно создавал или менял заказ, а рабочий режим превращался в свободную болталку.

## 2. Общие принципы

- По умолчанию рабочий бот должен оставаться безопасным.
- Локальный парсер остаётся основным безопасным путём для заказов.
- DeepSeek API не подключается на этом этапе.
- Ключи не читаются и не выводятся.
- Telegram polling не запускать.
- Рабочее поведение Telegram не менять.
- Все изменения кода будущих этапов должны идти только после отдельного задания.

## 3. Режим /заказ

Назначение:

строгий приём и разбор заказов.

Разрешено:

- принимать размеры;
- разбирать строки заказа;
- считать площадь;
- показывать preview;
- находить спорные строки;
- давать Corel-строки;
- показывать кнопки копирования.

Запрещено:

- свободно болтать;
- додумывать размеры;
- менять правила;
- создавать кодовые задачи;
- автоматически принимать спорные строки.

Если сообщение не похоже на заказ:

бот должен мягко предложить перейти в `/чат` или `/правила`, но не добавлять текст в заказ.

## 4. Режим /чат

Назначение:

свободное обсуждение работы, идей, бизнеса и общих вопросов.

Разрешено:

- обсуждать идеи;
- объяснять правила;
- помогать думать;
- предлагать улучшения;
- помогать сформулировать мысль;
- готовить черновик выжимки.

Запрещено:

- автоматически менять заказ;
- автоматически менять правила;
- автоматически менять код;
- запускать экспорт;
- считать разговор как заказ без явного перехода в `/заказ`.

Если в разговоре появилась полезная идея:

бот предлагает сделать `/выжимка`.

## 5. Режим /инженер

Назначение:

подготовка изменений проекта.

Разрешено:

- формировать задания для Codex/Hermes;
- проверять отчёты;
- готовить обновления для `MALYARKA_CURRENT_STATE.md`;
- планировать этапы;
- разделять задачи по риску.

Запрещено:

- менять код напрямую без Codex/Hermes;
- запускать API без разрешения;
- читать ключи;
- использовать `git add .`;
- делать commit/push без разрешения.

Инженерный режим должен работать по цепочке:

идея → выжимка → инженер → задание Codex/Hermes → проверка → принятие → обновление состояния → рабочий бот.

## 6. Режим /выжимка

Назначение:

превратить разговор или идею в краткий блок для инженера проекта.

Формат выжимки:

- что обсуждали;
- какая идея появилась;
- зачем это нужно;
- какой риск;
- что предлагается сделать;
- что не делать сейчас;
- готовый блок для инженера;
- при необходимости готовый блок для Codex/Hermes.

Запрещено:

- менять код;
- менять правила;
- автоматически добавлять идею в рабочий бот;
- считать выжимку принятой задачей без решения инженера.

## 7. Режим /правила

Назначение:

просмотр и проверка текущих правил проекта.

Разрешено:

- показывать краткое правило;
- объяснять правило;
- указывать источник;
- говорить, если правило не найдено;
- предлагать оформить новое правило через `/выжимка` или `/инженер`.

Запрещено:

- менять правила напрямую;
- выдумывать отсутствующее правило;
- использовать старые чаты как актуальный источник, если они противоречат `MALYARKA_CURRENT_STATE.md`.

## 8. Переключение режимов

Команды:

- `/заказ`
- `/чат`
- `/инженер`
- `/выжимка`
- `/правила`

Режим по умолчанию:

- После `/start` бот не должен автоматически считать следующий текст заказом.
- По умолчанию нужно показать меню выбора режима.
- Без выбранного режима бот должен просить выбрать действие, а не выполнять спорную интерпретацию сообщения.

После `/start` показывать меню:

- Новый заказ
- Чат
- Инженер
- Выжимка
- Правила

Как пользователь видит текущий режим:

- В ответах режима указывать короткий заголовок или статус, например `Режим: заказ`.
- В меню или кнопках показывать текущий активный режим.

Как выйти из режима:

- Команда `/start` возвращает к меню выбора режима.
- Будущая команда `/стоп` или `/выход` может сбрасывать текущий режим, если это будет принято отдельной задачей.

Что делать с сообщением, которое не подходит текущему режиму:

- В `/заказ`: не добавлять текст в заказ, если он не похож на заказ; предложить `/чат` или `/правила`.
- В `/чат`: не создавать заказ; предложить `/заказ`, если пользователь явно хочет оформить размеры.
- В `/инженер`: не менять код; предложить оформить задачу для Codex/Hermes.
- В `/выжимка`: просить исходную идею или разговор, если данных недостаточно.
- В `/правила`: говорить, что правило не найдено, и предлагать `/выжимка` или `/инженер`.

## 9. Безопасность

Общие запреты:

- не читать `.env`;
- не выводить токены и ключи;
- не читать `BOT_TOKEN`, `DEEPSEEK_API_KEY`, `GEMINI_API_KEY`;
- не трогать `orders.db` без отдельного задания;
- не трогать старый `bot.py`;
- не подключать реальный API;
- не запускать polling;
- не менять рабочее поведение Telegram на этом этапе.

## 10. Будущая интеграция DeepSeek

Пока DeepSeek API не подключается.

Будущий порядок:

1. Сначала режимы.
2. Потом state machine/router.
3. Потом DeepSeek только в `/чат`.
4. Потом `/выжимка`.
5. Потом RAG/Obsidian.
6. Потом Ollama.
7. Потом категоризатор.

## 11. Отложенная очередь идей

Отложенные направления, не текущие задачи:

- AI audit;
- Mock-режим для AI-тестов;
- `TASKS_QUEUE.md`;
- RAG/Obsidian;
- Ollama;
- категоризатор рабочего бота;
- проверка актуальных цен DeepSeek перед реальным подключением.

## 12. Что не делать сейчас

- не писать код режимов;
- не менять handlers;
- не подключать DeepSeek API;
- не добавлять ключи;
- не запускать polling;
- не менять рабочее поведение Telegram;
- не делать commit/push.
