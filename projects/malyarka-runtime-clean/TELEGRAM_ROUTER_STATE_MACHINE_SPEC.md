# TELEGRAM_ROUTER_STATE_MACHINE_SPEC

state_version: 0.8.2

## 1. Цель

Спроектировать безопасный router/state machine для одного Telegram-бота, чтобы режимы не смешивались:

- `/заказ`
- `/чат`
- `/инженер`
- `/выжимка`
- `/правила`

Главная цель:

не допустить, чтобы разговорный текст случайно стал заказом, а рабочий заказ попал в разговорный режим.

## 2. Режимы

### order

Команда: `/заказ`

Назначение:

строгий приём и разбор заказов.

Разрешено:

- принимать размеры;
- разбирать строки заказа;
- считать площадь;
- показывать preview;
- находить спорные строки;
- давать Corel-строки;
- показывать кнопки копирования.

Запрещено:

- свободно болтать;
- додумывать размеры;
- менять правила;
- менять код;
- автоматически принимать спорные строки.

### chat

Команда: `/чат`

Назначение:

свободное обсуждение работы, идей, бизнеса и общих вопросов.

Разрешено:

- обсуждать идеи;
- объяснять расчёты;
- объяснять правила;
- предлагать улучшения;
- помогать сформулировать мысль;
- готовить черновик выжимки.

Запрещено:

- автоматически создавать заказ;
- автоматически менять заказ;
- автоматически менять правила;
- автоматически менять код;
- запускать экспорт;
- считать разговор заказом без явного перехода в `/заказ`.

### engineer

Команда: `/инженер`

Назначение:

подготовка задач, проверок и изменений проекта.

Разрешено:

- формировать задания для Codex/Hermes;
- проверять отчёты;
- готовить обновления для `MALYARKA_CURRENT_STATE.md`;
- планировать этапы;
- разделять задачи по риску.

Запрещено:

- менять код напрямую без Codex/Hermes;
- запускать API без разрешения;
- читать ключи;
- использовать `git add .`;
- делать commit/push без разрешения.

### summary

Команда: `/выжимка`

Назначение:

подготовка краткой выжимки идеи для инженера проекта.

Разрешено:

- собрать суть разговора;
- выделить идею;
- описать пользу;
- описать риски;
- сформировать блок для инженера;
- при необходимости сформировать будущий блок для Codex/Hermes.

Запрещено:

- менять код;
- менять правила;
- автоматически добавлять идею в рабочий бот;
- считать выжимку принятой задачей без решения инженера.

### rules

Команда: `/правила`

Назначение:

просмотр и объяснение правил проекта.

Разрешено:

- показывать краткое правило;
- объяснять правило;
- указывать, что правило не найдено;
- предлагать оформить новое правило через `/выжимка` или `/инженер`.

Запрещено:

- менять правила напрямую;
- выдумывать отсутствующее правило;
- использовать старые чаты как актуальный источник, если они противоречат `MALYARKA_CURRENT_STATE.md`.

## 3. Где хранить текущий режим пользователя

Варианты:

1. In-memory storage: словарь `user_id -> current_mode`.
2. Позже база данных.
3. Позже отдельная таблица/хранилище сессий.

На первом этапе рекомендуется in-memory storage как безопасный минимальный вариант.

Причина:

- не трогаем `orders.db`;
- не меняем схему базы;
- не пишем реальные данные;
- легко тестировать;
- не ломаем текущий Telegram-бот.

Важно:

на этом этапе не трогать `orders.db` и не создавать новую базу.

## 4. Режим по умолчанию после /start

### Вариант A: order по умолчанию

Плюсы:

- быстрее для производства;
- пользователь сразу может присылать заказ.

Минусы:

- обычный разговор может быть ошибочно воспринят как заказ.

### Вариант B: neutral по умолчанию

Плюсы:

- безопаснее;
- бот сначала просит выбрать режим;
- разговор не попадёт случайно в заказ.

Минусы:

- на один клик больше перед заказом.

Рекомендация:

по умолчанию использовать `neutral` mode.

После `/start` бот должен показать меню:

- Новый заказ
- Чат
- Инженер
- Выжимка
- Правила

Заказ должен начинаться явно через `/заказ` или кнопку `Новый заказ`.

## 5. Переключение режимов

- `/заказ` включает `order` mode.
- `/чат` включает `chat` mode.
- `/инженер` включает `engineer` mode.
- `/выжимка` включает `summary` mode.
- `/правила` включает `rules` mode.

После переключения бот должен ответить:

`Включён режим: ...`

Например:

- `Включён режим заказа. Пришлите размеры.`
- `Включён режим чата. Можно обсуждать идеи и вопросы.`
- `Включён инженерный режим. Можно готовить задачу или выжимку.`
- `Включён режим выжимки. Пришлите текст идеи или разговора.`
- `Включён режим правил. Напишите, какое правило проверить.`

## 6. Обработка сообщений по режимам

### neutral mode

Если пользователь пишет обычный текст:

бот просит выбрать режим.

Если текст похож на заказ:

бот предлагает нажать `/заказ`, но не разбирает автоматически.

### order mode

Текст идёт в текущий локальный parser/preview.

Рабочее поведение Telegram должно остаться как сейчас.

### chat mode

Текст не идёт в parser заказа.

Текст считается разговором.

DeepSeek API пока не подключается.

### engineer mode

Текст не меняет код.

Текст используется для подготовки проектной задачи или плана.

### summary mode

Текст превращается в выжимку.

Выжимка не считается принятой задачей без решения инженера.

### rules mode

Текст используется для поиска/объяснения правил.

Правила не меняются напрямую.

## 7. Защита от смешивания

Обязательные правила:

- `chat` mode не создаёт заказ автоматически.
- `order` mode не ведёт свободную болталку.
- `engineer` mode не меняет код напрямую.
- `summary` mode не считается принятой задачей.
- `rules` mode не выдумывает отсутствующие правила.
- `neutral` mode не разбирает заказ автоматически без явного выбора режима.

Если сообщение похоже на заказ, но пользователь в `/чат`:

бот может предложить перейти в `/заказ`, но не должен автоматически создать заказ.

Если сообщение похоже на разговор, но пользователь в `/заказ`:

бот может предложить перейти в `/чат`, если заказ не распознан.

Если пользователь отправил фото:

photo handler остаётся заглушкой, пока фото-режим не подключён отдельным этапом.

## 8. Будущая структура модулей

Будущие файлы, которые предлагается рассмотреть на кодовом этапе, но не создавать сейчас:

- `malyarka_telegram/modes.py`
- `malyarka_telegram/router.py`
- `malyarka_telegram/session.py`
- `tests/test_malyarka_telegram_modes.py`
- `tests/test_malyarka_telegram_router.py`

### malyarka_telegram/modes.py

Хранит enum/константы режимов:

- `neutral`
- `order`
- `chat`
- `engineer`
- `summary`
- `rules`

### malyarka_telegram/session.py

Хранит минимальную in-memory session state:

- `user_id`
- `current_mode`

На первом этапе без базы данных.

### malyarka_telegram/router.py

Решает, куда направить входящее сообщение:

- в order parser;
- в chat placeholder;
- в engineer placeholder;
- в summary placeholder;
- в rules placeholder;
- в neutral menu.

### tests/test_malyarka_telegram_modes.py

Проверяет переключение режимов.

### tests/test_malyarka_telegram_router.py

Проверяет, что сообщения не смешиваются между режимами.
