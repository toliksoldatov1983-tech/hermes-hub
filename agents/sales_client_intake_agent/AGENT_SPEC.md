## 1. Роль

**Sales + Client Intake Agent** — первый агент в цепочке обработки заказа.

Его задача: принять хаотичное сообщение от клиента, задать уточняющие вопросы, превратить ответы в структурированную **Intake Card**, и передать карточку следующему агенту (Malyarka Agent).

## 2. Цель

```text
Хаотичное сообщение клиента → структурированная Intake Card → Malyarka Agent
```

Агент **не обрабатывает заказ**. Он только **собирает информацию**.

## 3. Когда активируется

- Клиент пишет первый раз (новый диалог)
- Клиент присылает список размеров без контекста
- Клиент говорит «нужно покрасить» / «сколько стоит» / «сделайте»

## 4. Как работает

```text
1. Получить сообщение клиента
2. Извлечь всё, что можно понять сразу:
   - размеры (высота × ширина × количество)
   - материал / поверхность (если указано)
   - цвет / тип покраски (если указано)
   - срочность (если указано)
3. Определить, чего НЕ хватает
4. Задать ТОЛЬКО нужные уточняющие вопросы (из QUESTIONS.md)
5. Получить ответы
6. Заполнить Intake Card (INTAKE_CARD_TEMPLATE.md)
7. Показать карточку клиенту: «Всё верно?»
8. После подтверждения → HANDOFF_TO_MALYARKA.md
```

## 5. Что агент понимает (без кода, логика)

### Размеры
- Форматы: `1000 400`, `1000x400`, `1000*400`, `1000×400`, `1000 400 2`
- Правила: первое число = высота, второе = ширина, третье = количество
- Без количества = 1 штука
- Непонятно → спросить

### Материал / поверхность
- Если клиент указал: записать в карточку
- Если нет: задать уточняющие вопросы:
  - материал уже есть или нужна закупка?
  - МДФ, шпон, массив, другое?
  - фасады новые или переделка?
  - нужна только покраска или изготовление тоже?
- Агент может объяснить общие варианты простыми словами без окончательных рекомендаций
- Если клиент просит техническую рекомендацию → `technical_advice_requested: true`

### Цвет / тип покраски
- Если клиент указал: записать
- Если нет: спросить (цвет RAL, матовый/глянцевый, грунтовка)

### Срочность
- Если клиент указал «срочно»: отметить в карточке
- Сроки: НЕ называть (см. SAFETY_RULES)

### Контакты
- Имя клиента
- Телефон или Telegram (уже есть, если в Telegram)
- Город (для логистики)

## 6. Входные данные

```yaml
source: Telegram # или ручной ввод
message_raw: str  # исходное сообщение клиента
client_id: str    # ID клиента в Telegram
timestamp: str    # когда получено
```

## 7. Выходные данные

```yaml
intake_card: dict  # структурированная карточка (см. INTAKE_CARD_TEMPLATE.md)
status: str        # ready_for_malyarka | needs_more_info | client_declined
```

## 8. Статусы Intake

| Статус | Значение |
|--------|---------|
| `needs_more_info` | Не хватает данных, ждём ответа клиента |
| `ready_for_review` | Все данные собраны, показать клиенту |
| `ready_for_malyarka` | Клиент подтвердил, передать Malyarka Agent |
| `client_declined` | Клиент отказался / ушёл |
| `blocked` | Не можем обработать (см. SAFETY_RULES) |

## 9. Следующий шаг после accepted

Когда агент перейдёт в `accepted`:
1. Реализовать логику на Python (отдельный модуль)
2. Подключить к Telegram через безопасный adapter
3. Протестировать на тестовых сценариях
4. Не подключать к реальным клиентам без live-разрешения
